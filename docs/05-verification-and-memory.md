# Verification and Memory

This document covers three load-bearing parts of Boris Cherny's loop methodology: how an agent verifies its own work, how repeated mistakes are turned into durable memory, and the permission model that lets loops run without a human approving every step.

## Self-verification: "can the agent actually run the thing?"

Cherny names verification as the single most important factor for getting good results out of Claude Code. His framing is that giving Claude a way to verify its own work will "2-3x the quality of the final result" ([howborisusesclaudecode.com](https://howborisusesclaudecode.com/)). The same point shows up in InfoQ's coverage of his workflow: "The most important tip is to give Claude a way to verify its work through a feedback loop, such as running a bash command, a test suite, or testing the app" ([InfoQ](https://www.infoq.com/news/2026/01/claude-code-creator-workflow/)).

The sharp version of the idea is that real verification means the agent can actually run and exercise what it changed, not just lint, type-check, or run unit tests. The agent needs some way to use the software it modified: Bash, an iOS or Android simulator, a desktop app, or computer use ([The Neuron](https://www.theneuron.ai/explainer-articles/claude-code-creators-boris-cherny-and-cat-wu-explain-how-to-use-agent-loops/)). For web work, Claude uses the Chrome extension to open a browser, test UI changes, and iterate until the change works, which is described as more token-efficient than Playwright or Chromium MCP servers ([howborisusesclaudecode.com](https://howborisusesclaudecode.com/)).

Cat Wu describes a concrete instance of this: a "desktop development skill" an engineer built that teaches Claude to launch the local desktop app, use computer use (computer vision) to click through the interface, exercise new UX, test edge cases, fix bugs, and recheck the result, then update the skill with what it learned. This attribution to Cat Wu comes from a secondary source (The Neuron) and is reported at medium confidence; the underlying capability that the desktop app can start and test web servers with built-in browser testing is corroborated on [howborisusesclaudecode.com](https://howborisusesclaudecode.com/).

Verification also extends to reading communication channels to diagnose environmental failures. When a build or test failure looks environmental rather than code-related, Claude is told to read Slack over MCP to self-diagnose: to check whether staging is down or whether a teammate already hit the same problem, before debugging further ([The Neuron](https://www.theneuron.ai/explainer-articles/claude-code-creators-boris-cherny-and-cat-wu-explain-how-to-use-agent-loops/)). The Slack-over-MCP diagnosis pattern is corroborated on Cherny's side, where he reads Slack via MCP to triage failing CI tests autonomously ([howborisusesclaudecode.com](https://howborisusesclaudecode.com/)). Cat Wu adds the scaling mechanism: "Synthesizing the memory from a lot of logs is a way to find these patterns more consistently" ([Every transcript](https://every.to/podcast/transcript-how-to-use-claude-code-like-the-people-who-built-it)).

For long-running loops, verification is wired in explicitly so the agent does not declare victory early. Cherny says that for very long-running tasks he will either prompt Claude to verify its work with a background agent when done, use an agent Stop hook to do that more deterministically, or use the ralph-wiggum plugin (originally by Geoffrey Huntley) ([Cherny, X thread](https://x.com/bcherny/status/2007179858435281082)). At Fortune Brainstorm Tech he described the payoff: "Many mornings I wake up, and Claude already has pull requests that it came up with, verified end to end, it has screenshots for me" ([Fortune](https://fortune.com/2026/06/11/anthropic-claude-boris-cherny-doesnt-write-code-by-hand-anymore/)).

## Memory and compounding: every mistake becomes a durable instruction

The mechanism that makes autonomous loops viable over time is that corrections are never left in chat. Cherny calls this the single most important idea for long-running work: "Every single time Claude makes a mistake, I don't tell it to do it differently. I tell it to write it to the CLAUDE.md, or make a skill, or something. If you can do this, then Claude can just run forever" ([howborisusesclaudecode.com](https://howborisusesclaudecode.com/)). In the Every podcast he puts it more plainly: "If you see a mistake, I'll just be like, at Claude, add this to Claude MD so that the next time it just knows this automatically" ([Every transcript](https://every.to/podcast/transcript-how-to-use-claude-code-like-the-people-who-built-it)).

This is the "compounding engineering" loop: correction becomes a durable instruction, which improves future runs. The Claude Code team shares one CLAUDE.md for the whole repo, checked into git and contributed to multiple times a week; Cherny tags @claude on PRs to have it update the file during review ([getpushtoprod](https://getpushtoprod.substack.com/p/how-the-creator-of-claude-code-actually)). His own shared CLAUDE.md is roughly 2.5k tokens and is updated whenever Claude makes a mistake ([note.com](https://note.com/ai_eng_tech/n/nf940a6dc47e6?hl=en)).

Frequent inner-loop workflows are captured the same way. Cherny uses slash commands for "every 'inner loop' workflow that I end up doing many times a day... Commands are checked into git and live in .claude/commands/" ([Cherny, X thread](https://x.com/bcherny/status/2007179847949500714)). Persisting both the rules (CLAUDE.md) and the procedures (skills and commands) into version control is what lets the whole team, and Claude, reuse the same fixes instead of relearning them each run.

This habit predates the loop methodology. Earlier in his career Cherny logged recurring code-review comments in a spreadsheet and, once a pattern recurred three or four times, wrote a lint rule to automate the comment away ([developing.dev](https://www.developing.dev/p/boris-cherny-creator-of-claude-code)). The CLAUDE.md/skill pattern is the same instinct applied to an agent.

## Permission model: plan mode to auto mode with safety classifiers

The earlier default was to start most tasks in plan mode. Cherny said he begins about 80% of his tasks in plan mode, iterates on the plan, then switches to auto-accept edits, because once the plan is good Claude "will one-shot the implementation almost every time" ([note.com](https://note.com/ai_eng_tech/n/nf940a6dc47e6?hl=en); [Pragmatic Engineer](https://newsletter.pragmaticengineer.com/p/building-claude-code-with-boris-cherny)).

He has since moved off plan mode as models improved: "I don't use [plan mode] anymore. I use auto mode... The newer models don't actually need a planning step. It was really important for Opus 4 through 4.5, but starting with 4.6, definitely with 4.7, it just doesn't need it" ([howborisusesclaudecode.com](https://howborisusesclaudecode.com/)). The shift is framed as model-capability-driven, not a claim that plan mode never helped.

Auto mode replaces per-action permission prompts with a model-based safety classifier: instead of approving individual commands or using `--dangerously-skip-permissions`, proposed actions are routed to a classifier, and safe ones are auto-approved ([howborisusesclaudecode.com](https://howborisusesclaudecode.com/)). Anthropic's engineering writeup describes a two-layer design: a server-side prompt-injection probe that scans tool outputs before they enter the agent's context, plus an output-layer transcript classifier using a fast single-token filter followed by reasoning only when flagged ([Anthropic Engineering](https://anthropic.com/engineering/claude-code-auto-mode)). Auto mode was released as an experimental capability, reported around March 24, 2026.

The argument for this is that manual approval stops being a real check once it becomes routine. Cherny: "When you accept 99% of requests, your eyes glaze over. Auto mode is more safe than reading every single permission prompt" ([howborisusesclaudecode.com](https://howborisusesclaudecode.com/)). The Neuron attributes a sharper version to Cat Wu: "If humans accept 99% of permission prompts, those prompts stop being meaningful" ([The Neuron](https://www.theneuron.ai/explainer-articles/claude-code-creators-boris-cherny-and-cat-wu-explain-how-to-use-agent-loops/)).

A sourcing caveat matters here. The "99%" is a rhetorical, rounded figure used in the interview. Anthropic's own engineering blog gives the measured number as 93%: "Claude Code users approve 93% of permission prompts" ([Anthropic Engineering](https://anthropic.com/engineering/claude-code-auto-mode)). Treat the underlying point as well-supported and the exact 99% as interview framing, not data. The attribution of the line to Cat Wu rather than Cherny is plausible but not confirmed against a primary transcript.

Auto mode was hardened empirically, not just by heuristics. The team collected thousands of full agent transcripts plus permission prompts and hardened the classifier through red-teaming ([howborisusesclaudecode.com](https://howborisusesclaudecode.com/)). Anthropic's engineering blog reports concrete eval datasets (real internal traffic n=10,000, real overeager actions n=52, synthetic data exfiltration n=1,000), a 0.4% false-positive and 17% false-negative rate, and an explicit caution that auto mode is "not a drop-in replacement for careful human review on high-stakes infrastructure" ([Anthropic Engineering](https://anthropic.com/engineering/claude-code-auto-mode)).

## How the three fit together

Verification is the inner feedback mechanism: the agent must be able to run, click through, and diagnose what it built. Memory is the compounding layer: each caught failure becomes a CLAUDE.md rule or a skill, so the same mistake does not recur and a loop can run unattended for long stretches. The permission model is the trust layer: a model-based classifier replaces rubber-stamped prompts, which is what makes it safe to let loops act without a human in front of every approval. Together they are the conditions under which Cherny's claim that "Claude can just run forever" becomes operationally true.
